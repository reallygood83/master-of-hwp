## 1. 한 줄 정의

**master-of-hwp**는 HWP/HWPX 문서를 AI가 읽고, 문맥을 이해하고, 사용자의 지시에 따라 문서를 수정하거나 새로 생성할 수 있게 만드는 AI 보조 HWP 워크플로우 프로젝트다.

---

## 2. 왜 지금 가능한가

이번 조사 기준으로 보면, 이 프로젝트는 **기술적으로 구현 가능**하다.

### 근거
1. **rhwp가 이미 HWP 문서 조작 기반을 갖고 있음**
   - HWP/HWPX 읽기/쓰기 가능
   - 편집 관련 액션 구조 보유
   - 표 같은 문서 요소 제어 가능성이 높음

2. **AI 연결 방식이 명확함**
   - Claude Code CLI, Codex CLI 모두 자동화/CLI/MCP 연동이 가능함
   - 별도 SaaS형 LLM API를 반드시 구매하지 않아도, 현재 구독 중인 도구를 활용하는 구조를 먼저 만들 수 있음

3. **MCP 방식이 특히 잘 맞음**
   - HWP 편집 기능을 MCP 서버의 tool로 감싸면, AI가 “문서를 읽고 → 판단하고 → 특정 편집 액션 호출”하는 패턴을 만들 수 있음

### 단, 현실적인 조건
아래는 바로 인정하고 시작해야 한다.

- **문서 완전 자유편집**은 초기부터 100% 자동화하기 어렵다.
- 특히 HWP의 **표, 스타일, 문단 속성, 개체, 그래프**는 텍스트 수정보다 난이도가 높다.
- 따라서 처음부터 “만능 AI 편집기”로 가기보다,
  **읽기 → 질의응답 → 제한된 수정 → 구조화된 생성** 순서로 올라가는 것이 맞다.

즉,

> **구현 가능하다. 다만 단계적으로 가야 한다.**

## 3. 최종적으로 만들고 싶은 모습

사용자는 HWP 파일을 열어두고 다음과 같이 말할 수 있다.

- “이 문서 핵심만 5줄로 요약해줘.”
- “3장 내용을 바탕으로 보고서용 문체로 다시 써줘.”
- “이 문서 내용을 참고해서 새 HWP 초안을 만들어줘.”
- “매출 현황 표를 추가해줘.”
- “이 문서 안에서 일정 관련 내용만 찾아서 정리해줘.”
- “이 문서를 기반으로 질의응답 가능하게 해줘.”

결과적으로 master-of-hwp는 아래 3가지를 제공하는 제품으로 발전할 수 있다.

1. **HWP 문서 대화형 분석**
2. **HWP 문서 AI 보조 편집**
3. **HWP 문서 AI 생성 및 템플릿 자동화**

## 4. 핵심 제품 시나리오

### 시나리오 A. 로컬 HWP 파일 대화형 분석
- 사용자가 로컬 HWP 파일을 선택
- 문서 내용을 구조적으로 추출
- 문단/섹션 단위로 쪼개어 인덱싱
- AI와 채팅하며 문서 기반 질문 가능

예시:
- “이 문서에서 예산 관련 내용만 모아줘.”
- “이 문서에서 표에 나온 수치를 설명해줘.”
- “2장과 4장의 논리 차이를 알려줘.”

### 시나리오 B. 문서 보조 편집
- AI가 문서 일부를 수정
- 선택 영역 기준 재작성
- 요약/확장/문체변환/맞춤법보정
- 특정 위치에 문단 삽입

예시:
- “선택한 문단을 더 공문서 스타일로 바꿔줘.”
- “이 문단을 반으로 줄여줘.”
- “이 목차 아래에 결론 문단 하나 추가해줘.”

### 시나리오 C. 표/구조 생성
- AI가 텍스트 명령을 표 구조로 바꿔 문서에 삽입
- 일정표, 비교표, 회의록 표, 예산표 등 템플릿화 가능

예시:
- “3행 4열 일정표 만들어줘.”
- “부서별 예산 비교표 넣어줘.”

### 시나리오 D. 새 HWP 문서 생성
- 프롬프트 기반으로 문서 초안 생성
- 기존 문서를 참고한 새 문서 생성
- 템플릿 기반 자동 작성

예시:
- “기존 사업계획서 톤을 유지해서 제안서 초안 만들어줘.”
- “회의록 형식으로 새 HWP 파일 생성해줘.”

### 시나리오 E. SaaS 확장
- 웹 기반 업로드/편집 UI 제공
- rhwp-studio 위에 AI 패널 탑재
- 팀 협업, 문서 버전관리, 템플릿 관리로 확장 가능

## 7. CLI형 vs SaaS형

## 7.1 CLI형

### 장점
- 지금 가진 구독 도구를 바로 활용 가능
- 초기 구축이 빠름
- 로컬 파일 워크플로우에 최적
- 보안 측면에서 유리할 수 있음

### 단점
- 일반 사용자 대상 제품화는 제한적
- UI/배포가 약함
- 설치/사용 진입장벽 존재

### 적합한 초기 목표
**가장 먼저 시작할 방식은 CLI형이 맞다.**

이유:
- 기술 검증 속도가 빠름
- 별도 서비스 인프라 부담이 적음
- 사용자 본인 워크플로우에 바로 적용 가능

## 7.2 SaaS형

### 장점
- 사용자 경험이 좋음
- 협업/템플릿/관리 기능으로 확장 가능
- 서비스 사업화 가능성 높음

### 단점
- 웹 에디터 통합 난이도 상승
- 파일 저장, 사용자 인증, 보안, 과금 등 고려 요소 증가
- API/배포/운영 체계 필요

### 결론
- **1단계는 CLI형**
- **2단계부터 SaaS형 프로토타입 검토**

## 8. 기술적으로 가장 중요한 질문에 대한 답

## 8.1 AI가 로컬 HWP 파일을 읽고 RAG처럼 대화할 수 있는가?

**가능하다.**

조건은 아래와 같다.
- HWP 문서에서 텍스트/구조를 추출할 수 있어야 함
- 문단/섹션 단위로 chunking 해야 함
- 해당 내용을 AI 컨텍스트에 주입하거나, 임베딩 기반 검색을 붙이면 됨

초기에는 굳이 무거운 RAG 시스템 없이도 가능하다.

### 초기형
- 문서 전체가 짧으면 그대로 AI에 전달

### 확장형
- 문단 단위 분할
- 질문과 관련된 chunk만 찾아 전달

즉, **“로컬 HWP 문서와 대화하기”는 1차 목표로 매우 적합하다.**

## 8.2 AI가 HWP 안의 내용을 수정할 수 있는가?

**가능하다.**

단계적으로 접근해야 한다.

### 쉬운 편
- 특정 텍스트 치환
- 문단 삽입
- 선택 영역 재작성
- 요약문 삽입

### 중간 난이도
- 제목 구조 반영
- 서식 일부 유지한 채 수정
- 위치 기반 삽입

### 높은 난이도
- 복잡한 표 레이아웃 조작
- 개체/도형/차트/고급 서식 정밀 제어

즉, 초기 버전에서는 **텍스트 중심 수정 + 제한적 구조 삽입**을 목표로 잡는 것이 현실적이다.

## 8.3 표나 그래프도 만들 수 있는가?

### 표
**표는 충분히 가능성이 높다.**

특히 다음 방식이 현실적이다.
- AI가 표 스키마 생성
- MCP tool이 표 생성 액션 호출
- 셀 데이터 채우기

예시:
- 일정표
- 비교표
- 회의록 표
- 예산표

### 그래프
그래프는 표보다 난도가 높다.

가능한 접근은 2가지다.
1. **우회 방식**: AI가 먼저 표 생성 → 차트는 외부 이미지/개체로 삽입
2. **직접 방식**: rhwp가 그래프/개체 삽입 제어를 지원하면 그 위에 MCP tool 추가

결론:
- **표는 1차 범위에 포함 가능**
- **그래프는 2차 이후 검토 항목**

## 9. 개발 전략

이번 프로젝트는 아래 순서로 가는 것이 가장 안전하다.

### Phase 1. 문서 읽기 검증
목표:
- HWP/HWPX 파일에서 텍스트와 구조를 안정적으로 추출한다.

산출물:
- 샘플 HWP 문서에서 텍스트 dump
- 문단/섹션 구조 JSON
- 표 추출 가능 여부 확인

완료 기준:
- “이 HWP 문서 내용을 AI가 읽고 설명할 수 있다” 수준 달성

### Phase 2. MCP 서버 최소 버전
목표:
- rhwp 기능을 AI가 tool 호출로 사용할 수 있게 만든다.

최소 tool 후보:
- 파일 열기
- 텍스트 추출
- 텍스트 치환
- 문단 삽입
- 새 파일 저장

완료 기준:
- Claude Code CLI 또는 Codex가 MCP를 통해 HWP 내용을 읽고 일부 수정 가능

### Phase 3. 대화형 문서 분석
목표:
- 로컬 HWP 기반 질의응답 구현

기능:
- 문서 요약
- 섹션 질의응답
- 키워드 추출
- 문서 비교

완료 기준:
- “이 문서 기반으로 질문하고 답받기” 가능

### Phase 4. 표 생성 자동화
목표:
- 사용자의 자연어를 표 구조로 변환해 HWP에 삽입

기능:
- 행/열 생성
- 셀 데이터 입력
- 기본 스타일 적용

완료 기준:
- “예산표 만들어줘” 수준의 명령이 실제 표 생성으로 이어짐

### Phase 5. 문서 생성 워크플로우
목표:
- 새 HWP 초안 자동 생성

기능:
- 템플릿 선택
- 문단 구조 자동 작성
- 기존 문서 기반 재작성

완료 기준:
- “회의록 HWP 새로 만들어줘” 가능

### Phase 6. SaaS 프로토타입 검토
목표:
- rhwp-studio 기반 웹 인터페이스로 확장

기능:
- 업로드
- AI 채팅 패널
- 미리보기/수정
- 저장

완료 기준:
- 브라우저 기반 데모 수준 구현

## 11. 기술 선택 제안

## 11.1 추천 1차 스택

### 문서 엔진
- rhwp

### AI 실행기
- Claude Code CLI 우선
- Codex CLI 보조

- **1순위: Python FastMCP**
  - 빠르게 만들기 좋음
  - 프로토타입 속도 유리

- **2순위: TypeScript MCP SDK**
  - 향후 웹 확장과의 연결성 좋음

### 저장/색인
- 초기: 로컬 JSON/SQLite
- 확장: 벡터 DB

### UI
- 초기: CLI
- 확장: rhwp-studio

## 11.2 왜 Python FastMCP를 먼저 추천하는가
- 도구 래핑이 빠름
- 프로토타이핑 속도가 좋음
- MCP 최소 기능 구현이 쉬움

즉,

> **첫 구현은 Python MCP 서버 + Claude Code CLI + rhwp 조합이 가장 빠르다.**

## 12. 주요 리스크

### 리스크 1. rhwp 액션의 실제 제어 범위 확인 필요
README/문서만으로는 충분하지 않고, 실제 코드/샘플 파일로 검증이 필요하다.

대응:
- 먼저 “텍스트 추출 / 텍스트 삽입 / 표 생성” 3개만 PoC 검증

### 리스크 2. HWP 서식 보존 문제
AI가 재작성한 텍스트를 넣을 때 서식이 깨질 수 있다.

대응:
- 초기에는 plain text 수준의 안정성 확보
- 이후 스타일/문단 속성 보존 연구

### 리스크 3. 표/개체 조작 난이도
표는 가능성이 높지만, 모든 형태를 자연어 하나로 완벽히 만들기는 어렵다.

대응:
- “정형화된 표 생성”부터 시작
- 자유도보다 성공률 우선

### 리스크 4. 그래프 기능의 구현 범위
그래프는 지원 구조에 따라 난도가 급상승할 수 있다.

대응:
- MVP 제외
- 후속 기능으로 이동

### 리스크 5. CLI 의존성

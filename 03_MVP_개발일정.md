# master-of-hwp MVP 개발일정

- 작성일: 2026-04-19
- 프로젝트: master-of-hwp
- 목적: 기획안, 기술검증 체크리스트, MCP 서버 설계서를 실제 구현 순서로 번역하여 **바로 실행 가능한 MVP 개발 일정**으로 정리한다.

---

## 1. 이번 MVP의 목표

이번 MVP는 거창한 완제품을 만드는 단계가 아니다.
핵심은 아래 3가지를 실제로 증명하는 것이다.

1. **HWP/HWPX 문서를 AI가 읽을 수 있다**
2. **AI가 문서 기반 질문응답과 제한적 수정을 할 수 있다**
3. **공무원/교사 실무에 바로 연결되는 표 생성과 문서 보조가 가능하다**

즉,

> **이번 MVP의 성공 기준은 “된다”가 아니라, “실제로 써볼 만하다”를 만드는 것이다.**

---

## 2. MVP 범위 재정의

### 이번 MVP에 포함
- HWP/HWPX 문서 열기
- 문서 전체 텍스트 추출
- 문서 구조 추출
- 문서 기반 요약/질문응답
- 특정 문단 치환
- 특정 문단 뒤 새 문단 삽입
- 간단한 표 생성
- 새 파일로 저장
- 저장 결과 검증

### 이번 MVP에서 제외
- 고급 그래프 생성
- 복잡한 도형/개체 조작
- 완벽한 스타일 보존
- 웹 SaaS UI
- 계정/권한/협업 기능

---

## 3. 일정 운영 원칙

- 기능은 **읽기 → 구조 → 수정 → 저장 → 표 생성** 순서로 쌓는다.
- 매 단계마다 **실행 가능한 데모**가 있어야 한다.
- 다음 단계는 이전 단계 검증이 끝나야 시작한다.
- 문서 작성만 하지 않고, 항상 **샘플 파일 기준 검증 결과**를 남긴다.

---

## 4. 전체 일정 개요

이번 MVP는 5단계로 진행한다.

1. **Phase 0. 환경/샘플 준비**
2. **Phase 1. 읽기 PoC**
3. **Phase 2. 수정 PoC**
4. **Phase 3. 표 생성 PoC**
5. **Phase 4. 역할 특화 데모 정리**

시간 기준으로 보면,
집중해서 진행할 경우 **약 1~2주짜리 압축 MVP**로 볼 수 있다.

## 6. Phase 1 — 읽기 PoC

### 목표
AI가 HWP/HWPX 문서를 읽고 이해할 수 있는 기반을 완성한다.

### 핵심 구현 대상
- `open_document`
- `extract_document_text`
- `extract_document_structure`

### 작업 항목
- [x] rhwp 래퍼 초안 작성 (`mcp-server/bridges/rhwp_extract.mjs` + `adapters/rhwp_adapter.py`)
- [x] 파일 열기 로직 작성 (`tools/open_document.py`)
- [x] 전체 텍스트 추출 로직 작성 (`_extract_with_rhwp` → Node 브리지)
- [x] 문단/섹션 구조 추출 로직 작성 (`extract_structure` / `_sections_from_bridge_paragraphs`)
- [ ] 표 존재 여부 감지 로직 작성 (MD 휴리스틱만 있음, HWPX 표 감지 예정)
- [ ] 샘플 문서 기준 추출 결과 확인 (외부 실제 .hwp/.hwpx 샘플 확보 필요)

### 검증 질문
- [ ] 문서 전체가 정상적으로 읽히는가
- [ ] 한글 깨짐 없이 출력되는가
- [ ] 제목/본문/표 구분이 어느 정도 되는가
- [ ] 긴 문서에서도 추출 실패가 없는가

### 산출물
- 텍스트 추출 가능한 최소 MCP 서버 초안
- 구조 JSON 예시 파일
- 읽기 검증 로그

### 완료 기준
- 공무원 샘플 문서 1개, 교사 샘플 문서 1개에서
  텍스트 및 구조 추출이 모두 성공한다.

### 권장 데모
- “이 문서 핵심 요약해줘.”
- “이 문서의 표가 몇 개 있는지 알려줘.”

## 7. Phase 2 — 수정 PoC

### 목표
AI가 문서를 읽는 것에서 끝나지 않고,
특정 위치를 수정할 수 있음을 증명한다.

### 핵심 구현 대상
- `replace_paragraph_text`
- `insert_paragraph_after`
- `save_as`
- `validate_document`

### 작업 항목
- [ ] 문단 인덱스 기반 수정 로직 작성
- [ ] 문단 삽입 로직 작성
- [ ] 수정본 저장 로직 작성
- [ ] 저장 후 재검증 로직 작성
- [ ] 원본/수정본 비교 메모 남기기

### 검증 질문
- [ ] 특정 문단만 안정적으로 바꿀 수 있는가
- [ ] 문단 삽입 위치가 어긋나지 않는가
- [ ] 수정 후 파일이 깨지지 않는가
- [ ] 저장본을 다시 읽었을 때 구조가 유지되는가

### 산출물
- 수정 가능한 MCP 서버 버전
- 수정 전/후 샘플 결과 파일
- 문단 수정 테스트 로그

### 완료 기준
- 최소 2개 샘플 문서에서 문단 치환/삽입/저장이 모두 성공한다.

### 권장 데모
- 공무원 문서: “이 문단을 더 공문서 스타일로 바꿔줘.”
- 교사 문서: “이 안내문 마지막에 학부모 협조 문구를 추가해줘.”

## 8. Phase 3 — 표 생성 PoC

### 목표
공무원/교사 업무에서 반복 사용 가치가 높은 표 생성 기능을 붙인다.

### 핵심 구현 대상
- `create_table`
- `fill_table_cells`

### 우선 표 유형
- 공무원용 일정표
- 공무원용 예산 비교표
- 교사용 상담 기록표
- 교사용 행사 운영 체크표

### 작업 항목
- [ ] 기본 빈 표 생성
- [ ] 행/열 지정 처리
- [ ] 헤더 포함 여부 처리
- [ ] 셀 입력 로직 작성
- [ ] 문서 내 삽입 위치 처리
- [ ] 샘플 표 생성 결과 확인

### 검증 질문
- [ ] 표가 실제로 생성되는가
- [ ] 셀 데이터가 정확한 위치에 들어가는가
- [ ] 간단한 정형 표는 다시 손볼 필요가 적은가
- [ ] 공무원/교사 시나리오에 바로 적용 가능한가

### 산출물
- 표 생성 가능한 MCP 서버 버전
- 역할별 표 샘플 결과 파일
- 표 생성 테스트 로그

### 완료 기준
- 최소 2종 이상의 업무형 표 생성이 성공한다.

### 권장 데모
- “부서별 예산 비교표 만들어줘.”
- “학생 상담 기록표 만들어줘.”

## 9. Phase 4 — 역할 특화 데모 정리

### 목표
기술이 아니라 제품 가치 관점에서,
공무원/교사 각각에게 의미 있는 데모를 만든다.

### 공무원 데모 목표
아래 중 최소 1개는 완성도 있게 보여줄 수 있어야 한다.

- [ ] 기존 보고서 요약 + 공문체 재작성
- [ ] 회의자료 핵심 정리 + 표 생성
- [ ] 사업계획 문서 일부 자동 정리

### 교사 데모 목표
아래 중 최소 1개는 완성도 있게 보여줄 수 있어야 한다.

- [ ] 가정통신문 초안 보정
- [ ] 수업안 구조 정리
- [ ] 학생 상담 기록표 생성

### 작업 항목
- [ ] 공무원 데모 스크립트 작성
- [ ] 교사 데모 스크립트 작성
- [ ] 데모 입력 문서 고정
- [ ] 데모 결과 파일 저장
- [ ] 데모 전후 비교 설명 작성

### 산출물
- 공무원용 데모 1종
- 교사용 데모 1종
- 시연용 스크립트

### 완료 기준
- 제3자에게 보여줘도 “이건 실무에 쓸 수 있겠다”는 반응이 나오는 수준

## 12. 리스크 관리 일정

### 리스크 1. 구조 추출 불안정
**발생 시점:** Phase 1

대응:
- 구조 추출이 불안정하면,
  1차 수정 기능은 문단 인덱스 기반 최소 범위만 지원한다.

### 리스크 2. 저장 후 파일 손상
**발생 시점:** Phase 2

대응:
- 저장 직후 `validate_document`를 필수 단계로 넣는다.
- 원본 덮어쓰기는 금지한다.

### 리스크 3. 표 생성 API 한계
**발생 시점:** Phase 3

대응:
- 병합 셀/고급 서식은 제외하고,
  단순 표부터 성공시킨다.

### 리스크 4. 공무원/교사 특화 가치가 약함
**발생 시점:** Phase 4

대응:
- 기능 중심이 아니라, 실제 문서 결과물 중심으로 데모를 재구성한다.

# master-of-hwp 기술검증 체크리스트

- 작성일: 2026-04-19
- 프로젝트: master-of-hwp
- 목적: rhwp 기반 HWP AI 보조 시스템의 **기술 실현 가능성**, **우선 구현 범위**, **공무원/교사 특화 기능 방향**을 단계적으로 검증하기 위한 체크리스트

---

## 1. 이 문서의 역할

이 문서는 바로 개발에 들어가기 전에,
무엇을 먼저 확인해야 하고,
어디까지 되면 “된다”고 판단할 수 있는지를 정리한 문서다.

핵심은 3가지다.

1. **정말 구현 가능한가**
2. **어디까지를 1차 목표로 잡아야 하는가**
3. **공무원과 교사에게 실제로 쓸모 있는 기능은 무엇인가**

---

## 2. 검증 원칙

이번 프로젝트는 아래 원칙으로 검증한다.

- 큰 기능보다 **작은 성공**을 먼저 만든다.
- 문서 전체 자동화보다 **읽기 → 질문응답 → 제한적 수정**을 먼저 본다.
- 화려한 기능보다 **반복 사용 가치**를 우선 본다.
- 공무원/교사 대상 기능은 “있으면 좋은 기능”이 아니라, **실제 업무 시간을 줄이는 기능** 위주로 본다.

## 5. 1차 검증: HWP 읽기 가능 여부

목표:
AI가 문서를 이해하기 위한 가장 기본 단계로,
HWP/HWPX의 텍스트를 안정적으로 읽을 수 있는지 확인한다.

### 체크리스트
- [ ] 샘플 HWP 파일 1개에서 전체 텍스트 추출 가능
- [ ] 샘플 HWPX 파일 1개에서 전체 텍스트 추출 가능
- [ ] 문단 순서가 원문과 크게 다르지 않음
- [ ] 특수문자/한글이 깨지지 않음
- [ ] 페이지 수가 많은 문서도 추출 실패 없이 처리됨
- [ ] 표 내부 텍스트도 최소한 읽어올 수 있음

### 성공 기준
- 사용자가 “이 문서 무슨 내용이야?”라고 물으면,
  AI가 문서 전체 흐름을 이해하고 답할 수 있을 정도의 텍스트 추출이 가능해야 한다.

### 실패 시 체크 포인트
- [ ] 텍스트 누락 구간이 있는가
- [ ] 표 내부 내용이 빠지는가
- [ ] 문단 순서가 섞이는가
- [ ] 한글 인코딩 이슈가 있는가

---

## 6. 2차 검증: 문서 구조 추출 가능 여부

목표:
단순 텍스트가 아니라,
문서의 제목/문단/표/섹션 구조를 어느 정도 파악할 수 있는지 확인한다.

### 체크리스트
- [ ] 제목과 본문을 구분할 수 있음
- [ ] 문단 단위 배열 추출 가능
- [ ] 표 위치를 식별 가능
- [ ] 표의 행/열 수를 식별 가능
- [ ] 문서 내 특정 위치(예: 3번째 문단, 2번째 표)를 지정할 수 있음
- [ ] 향후 수정에 필요한 위치 정보 모델 설계 가능

### 성공 기준
- AI가 “2장 아래 본문만 다시 써줘” 같은 요청을 처리할 수 있는 수준의 구조 정보가 확보되어야 한다.

### 실패 시 체크 포인트
- [ ] 제목 구조를 잃어버리는가
- [ ] 문단 인덱싱이 불안정한가
- [ ] 표와 본문이 뒤섞이는가

---

## 7. 3차 검증: HWP 기반 AI 질의응답

목표:
로컬 HWP 문서를 AI가 읽고,
문서 기반 질문에 답할 수 있는지 확인한다.

### 기본 질문 예시
- “이 문서 핵심을 5줄로 요약해줘.”
- “이 문서에서 일정 관련 내용만 정리해줘.”
- “예산 관련 항목만 뽑아줘.”
- “이 문서의 목적과 결론을 설명해줘.”

### 체크리스트
- [ ] 문서 전체 요약 가능
- [ ] 특정 키워드 중심 답변 가능
- [ ] 문서 내 특정 섹션 인용 가능
- [ ] 표 내용을 참고한 답변 가능
- [ ] 긴 문서에서도 질문응답 품질이 크게 무너지지 않음
- [ ] 잘 모르는 내용은 문서에 없다고 말할 수 있음

### 성공 기준
- 사용자가 문서를 열어놓고 실제로 대화하듯 사용할 수 있어야 한다.

### 추가 판단 포인트
- [ ] 단순 컨텍스트 주입만으로 충분한가
- [ ] chunking 기반 경량 RAG가 필요한가
- [ ] 임베딩 저장소 도입이 필요한가

---

## 8. 4차 검증: 문서 수정 가능 여부

목표:
AI가 문서를 단순히 읽는 것을 넘어서,
지정된 위치의 내용을 안전하게 수정할 수 있는지 확인한다.

### 검증할 수정 유형
- [ ] 특정 문단 치환
- [ ] 문단 삽입
- [ ] 선택 영역 요약본 삽입
- [ ] 선택 영역 문체 변환
- [ ] 오탈자/표현 개선

### 테스트 명령 예시
- “이 문단을 더 공식적인 문체로 바꿔줘.”
- “이 문단을 3줄로 줄여줘.”
- “결론 문단 하나 추가해줘.”
- “맞춤법만 다듬어줘.”

### 체크리스트
- [ ] 텍스트 치환 가능
- [ ] 지정 위치 삽입 가능
- [ ] 수정 후 저장 가능
- [ ] 문서가 깨지지 않음
- [ ] 문단 순서/구조 유지
- [ ] 원본과 수정본 비교 가능

### 성공 기준
- “AI가 문서를 조금 수정해주는 수준”이 실제로 안정적으로 작동해야 한다.

### 실패 시 체크 포인트
- [ ] 저장 후 파일이 깨지는가
- [ ] 수정 위치가 어긋나는가
- [ ] 문서 서식이 과도하게 깨지는가

---

## 9. 5차 검증: 표 생성 가능 여부

목표:
공무원/교사 업무에서 자주 쓰는 표를 자연어로 생성할 수 있는지 확인한다.

### 우선 검증할 표 유형
- [ ] 일정표
- [ ] 비교표
- [ ] 예산표
- [ ] 학생 관찰 기록표
- [ ] 회의 정리표

### 체크리스트
- [ ] 빈 표 생성 가능
- [ ] 행/열 지정 가능
- [ ] 셀 텍스트 입력 가능
- [ ] 간단한 헤더 구조 생성 가능
- [ ] 표 생성 후 문서 저장 가능
- [ ] 본문 뒤 특정 위치 삽입 가능

### 테스트 명령 예시
- “4행 3열 일정표 만들어줘.”
- “부서별 예산 비교표 넣어줘.”
- “학생 상담 기록용 표 만들어줘.”

### 성공 기준
- 단순한 정형 표는 사람이 손으로 다시 만들 필요 없을 정도여야 한다.

### 그래프 관련 판단
- [ ] 그래프/차트는 현재 범위에서 제외할지 결정
- [ ] 필요하면 표 → 차트 이미지 삽입 우회안을 검토

목표:
Claude Code CLI 또는 Codex CLI가 rhwp 기능을 tool처럼 호출할 수 있게 만든다.

### 최소 MCP tool 후보
- [ ] `extract_document_text`
- [ ] `extract_document_structure`
- [ ] `replace_text`
- [ ] `insert_paragraph`
- [ ] `create_table`
- [ ] `save_as`

### 체크리스트
- [ ] MCP 서버 실행 가능
- [ ] CLI에서 MCP 서버 인식 가능
- [ ] tool 목록 정상 노출
- [ ] tool 호출 결과 JSON 구조 안정적
- [ ] 에러 메시지 해석 가능
- [ ] 잘못된 요청 시 안전하게 실패함

### 성공 기준
- 사용자가 자연어 요청을 하면,
  AI가 적절한 MCP tool을 호출해 실제 문서 조작까지 이어져야 한다.

### 실패 시 체크 포인트
- [ ] tool 스키마가 불명확한가
- [ ] 응답 구조가 AI가 쓰기 어렵게 되어 있는가
- [ ] 문서 위치 지정 방식이 모호한가

## 11. 공무원 특화 기능 검증 트랙

공무원 대상 기능은 “문장 잘 써주는 기능”보다,
**반복 행정업무를 줄여주는 기능**이 핵심이다.

## 11.1 공무원 핵심 사용 시나리오
- [ ] 공문 초안 작성 보조
- [ ] 보고서 요약 및 문체 정리
- [ ] 회의자료 정리
- [ ] 사업계획서 초안 생성
- [ ] 예산 비교표/업무 일정표 생성
- [ ] 기존 문서 기반 재작성

## 11.2 공무원 특화 기능 후보
- [ ] 공문서 문체 변환
- [ ] 보고서용 요약 자동 생성
- [ ] 항목별 정리(배경/목적/추진내용/기대효과)
- [ ] 회의록 정리 포맷 자동화
- [ ] 사업계획서 초안 템플릿 생성
- [ ] 예산표 자동 생성

## 11.3 공무원 검증 질문
- [ ] “기존 보고서를 바탕으로 공문 초안 생성”이 가능한가

- [ ] 표 생성 기능이 행정 문서 작성에 바로 도움이 되는가

## 11.4 공무원 MVP 후보 기능
우선순위가 높은 기능:
- [ ] 문서 요약
- [ ] 공문서 문체 변환

- [ ] 회의자료 표 생성

## 13. 공무원/교사 공통으로 검증할 포인트

## 14. 보안 및 개인정보 체크리스트

초기부터 꼭 점검해야 한다.

- [ ] 로컬 파일만 처리하는 모드 제공 가능
- [ ] 외부 전송 여부를 명확히 설명 가능
- [ ] 테스트용 샘플 문서는 개인정보 비식별 처리
- [ ] 학생 정보/민원 정보/행정자료 등 민감정보 문서 분리 관리
- [ ] 로그에 문서 원문이 과도하게 남지 않도록 설계

### 체크리스트
- [ ] 로컬에서 실행 절차가 복잡하지 않음
- [ ] HWP 파일 지정 방식이 직관적임
- [ ] 문서 읽기/수정 명령 흐름이 자연스러움
- [ ] 결과 파일 위치를 쉽게 확인할 수 있음
- [ ] 오류가 났을 때 사용자가 원인을 이해할 수 있음

### 판단 기준
- 개발자 본인이 매일 쓸 수 있어야 한다.
- “가능하다”가 아니라 “계속 쓰고 싶다” 수준이어야 한다.

## 16. SaaS 확장 검토 체크리스트

초기에는 CLI가 맞지만,
공무원/교사 대상 확장을 생각하면 웹형도 장기적으로 의미가 있다.

### 검토 항목
- [ ] rhwp-studio에 AI 패널 추가 가능한가
- [ ] 웹 업로드/미리보기/저장 흐름 구성 가능한가
- [ ] 사용자별 문서 저장 구조 설계 가능한가
- [ ] 문서 버전관리 필요성이 큰가
- [ ] 기관용 배포 모델이 필요한가

### SaaS로 갈 이유가 생기는 조건
- [ ] 팀 단위 사용 수요가 있음
- [ ] 반복 템플릿 관리 수요가 큼
- [ ] 기관 내부 배포 니즈가 있음
- [ ] 협업/승인 흐름까지 확장 가능함
